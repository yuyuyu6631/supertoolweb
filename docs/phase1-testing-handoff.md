# 一期转测试说明：统一入口 + 双模式搜索 + 轻量 AI 搜索承接

## 一、当前结论

一期开发先到这里，**本版本不继续加功能，直接转测试**。

这一版的定位不是做完整 AI 搜索产品，也不是做多轮聊天助手，而是完成下面这件事：

**把首页统一入口跑通，让 `直接搜索` 和 `AI 帮找` 两种模式都能稳定进入 `/tools` 承接，并让 AI 模式具备基础的“理解需求、缩小范围、展示理由”能力。**

一句话说：

**这一版的目标是“可测、可用、可降级”，不是“功能继续扩张”。**

## 二、本版测试目标

本轮测试重点不是验证 AI 有多聪明，而是验证以下四件事：

### 1. 入口是否统一

首页不再同时承担“普通搜索 + 大聊天框 + 悬浮聊天”的多入口逻辑，而是统一为一个主输入入口。

### 2. 双模式链路是否可用

用户从首页进入 `/tools` 后，`search` 和 `ai` 两种模式都能正常承接。

### 3. AI 模式是否稳定

AI 模式能返回结构化理解面板、推荐理由、快捷动作，并在异常情况下正常降级。

### 4. 现有目录能力是否未被破坏

原有目录搜索、筛选、列表渲染、详情跳转等核心流程不应被这次改动影响。

## 三、测试范围冻结

本轮转测试后，以下范围视为**一期冻结范围**，不再追加新功能：

### 已纳入测试范围

1. 首页统一入口
2. `直接搜索 / AI 帮找` 模式切换
3. `/tools?mode=search|ai&q=...` 路由承接
4. `/api/ai-search` 聚合接口
5. AI 理解面板展示
6. 快捷动作映射真实筛选
7. 工具卡推荐理由展示
8. AI 模式失败降级回目录能力
9. 首页悬浮聊天入口隐藏策略
10. 非首页轻量浮动入口保留

### 明确不在本轮测试范围

1. 多轮会话体验
2. AI 抽屉化重构
3. 新推荐接口或新推荐算法
4. 向量检索 / RAG / rerank
5. 大模型生成长推荐文案
6. 详情页信息架构调整
7. 新主题或全站视觉重构
8. 二期交互增强能力

## 四、本版已实现内容

## 4.1 后端：新增 AI 搜索聚合接口

已新增：

- 路由：`/api/ai-search`
- 路由文件：`ai_search.py`
- 主服务：`ai_search_service.py`
- 返回 schema：`ai_search.py`
- 路由注册：`router.py`

### 已实现能力

1. Query 轻量规范化  
   包括空白处理、词映射、停用词清理

2. 检索与意图解析并行执行  
   检索和 LLM 解析不互相阻塞

3. LLM 只做结构化 JSON 意图解析  
   不做长文本聊天输出

4. LLM 失败自动降级  
   不阻塞主列表返回

5. Redis 意图缓存  
   热门词长 TTL，普通词短 TTL

6. 统一返回结构  
   返回 `ai_panel + results + directory + meta`

7. 每个结果补充模板化 `reason`  
   用于表达“为什么推荐”，但不改变主搜索逻辑

## 4.2 前端：AI 模式接入真实接口

已完成：

- `mode=ai` 改为调用 `/api/ai-search`
- 页面入口改造：`page.tsx`
- API 封装新增：`fetchAiSearch`
- 类型补充：`AiSearchResponse / AiPanel / QuickAction` 等

### 页面能力

1. AI 面板改为真实数据驱动
2. 展示以下内容：
   - 你的需求
   - 系统理解
   - 当前逻辑
   - 快捷动作
3. 快捷动作映射真实 URL 筛选
4. AI 模式接口失败时，自动降级为目录接口
5. 工具卡增加推荐理由展示

## 4.3 组件展示调整

已完成：

- `ToolCard.tsx` 新增 `reason` 展示
- `CompareToolsGrid.tsx` 支持透传 `reason`

注意：

**这里的 `reason` 是模板化推荐提示，不是完整 AI 推理结论。**

## 五、测试重点

下面这部分是本轮测试最该盯的。

## 5.1 首页入口链路

测试确认：

1. 首页默认模式是否为 `直接搜索`
2. 切换到 `AI 帮找` 后，输入内容是否保留
3. 按钮文案是否随模式变化
4. 回车提交与按钮点击是否一致
5. 示例词点击是否遵循当前模式提交
6. 首页是否已移除内嵌大聊天区
7. 首页是否不再展示可展开悬浮聊天入口

## 5.2 `/tools` 承接逻辑

测试确认：

1. `mode=search` 时走目录搜索视图
2. `mode=ai` 时展示 AI 理解面板
3. 非法 `mode` 是否自动回退为 `search`
4. 刷新页面后 `mode` 和 `q` 是否保留
5. 浏览器前进 / 后退后状态是否一致
6. URL 是否为唯一状态源，而不是本地 state 漂移

## 5.3 `/api/ai-search` 能力验证

测试确认：

1. 正常 query 返回 200
2. 返回数据结构完整
3. `ai_panel` 字段可解析
4. `results` 列表可正常渲染
5. `reason` 字段存在时正常展示
6. 缓存命中与未命中都不影响正确性
7. LLM 失败时，接口是否仍能给出可用列表
8. Redis 异常时，接口是否仍可用
9. 非法或极端 query 是否不会打崩接口

## 5.4 AI 面板与快捷动作

测试确认：

1. `mode=ai` 下 AI 面板是否出现
2. `mode=search` 下 AI 面板是否不出现
3. “你的需求”是否正确展示原始 query
4. “系统理解”是否为结构化理解，不是空白或脏数据
5. “当前逻辑”是否与筛选行为一致
6. 快捷动作点击后是否真实更新 URL
7. URL 更新后，结果列表是否同步变化
8. 多次点击快捷动作后，状态是否仍一致
9. “进入筛选列表”是否表现正常

## 5.5 降级与容错

这是本轮必须重点测的部分。

测试确认：

1. AI 接口失败时，列表是否仍可展示
2. AI 面板缺失时，页面是否不会崩
3. LLM 返回非法 JSON 时，是否走默认降级文案
4. 检索成功但解析失败时，页面是否仍可用
5. 检索为空时，页面是否能显示正常空态
6. 慢请求是否不会导致页面永远 loading
7. 接口超时是否能安全回退

## 5.6 回归测试

确认以下旧能力不被影响：

1. 目录筛选
2. 排序
3. 分页
4. 工具卡渲染
5. 对比链路
6. 工具详情页跳转
7. 非首页浮动入口
8. 现有搜索目录接口能力

## 六、验收标准

本版通过测试的标准不是“AI 理解完美”，而是满足以下条件：

### P0 必过

1. 首页只有一个主入口，不再重复
2. `search / ai` 两种模式都能稳定进入 `/tools`
3. `mode=ai` 页面能正常展示 AI 理解面板
4. AI 接口失败时页面不崩，能正常降级
5. 原有目录搜索与筛选不回归

### P1 建议通过

1. 快捷动作与 URL 筛选联动稳定
2. 推荐理由展示正常
3. 刷新 / 前进 / 后退状态一致
4. 首页与非首页的浮动入口行为符合预期

## 八、已知限制

这部分要提前说清，不然后面容易把“一期限制”误判成 bug。

### 当前限制 1：AI 不是完整聊天搜索

这一版的 AI 只是轻量理解层，不是多轮会话助手。

### 当前限制 2：推荐理由是模板化的

`reason` 用来增强解释感，不代表模型深度判断。

### 当前限制 3：搜索结果仍以现有目录逻辑为主

AI 不直接改写底层搜索引擎，只负责理解和补充筛选建议。

### 当前限制 4：AI 模式和普通搜索在数据层可能仍高度复用

一期重点是承接和表达差异，不是完全重构推荐排序。

## 九、风险说明

### 风险 1：AI 感知可能偏弱

因为一期没有做深推荐链路，用户可能感知到的差异主要来自 AI 面板和快捷动作。

### 风险 2：极端 query 下解析不稳定

这类问题本轮应以“可降级、不崩”为验收原则，而不是要求所有 query 都完美理解。

### 风险 3：缓存 / LLM / Redis 异常可能暴露边界问题

所以测试时应重点压异常路径，而不是只测正常路径。

## 十、回滚策略

如果本轮测试发现高风险问题，本版支持快速回滚。

### 可回滚范围

1. 前端 `mode=ai` 请求切回现有目录接口
2. AI 面板渲染开关关闭
3. 首页入口回退为单一搜索模式
4. `/api/ai-search` 接口隔离下线，不影响原目录接口

### 回滚原则

**优先保主搜索链路，不保 AI 展示层。**
