# 08 技术架构

## 1. 架构结论

当前仓库的真实架构基线如下：

| 层级 | 当前采用 |
| --- | --- |
| 仓库结构 | NPM Workspace Monorepo |
| 前端框架 | Next.js 15 App Router + React 19 |
| 后端框架 | FastAPI |
| 数据层 | MySQL + SQLAlchemy 2 + Alembic |
| 样式层 | Tailwind CSS 4 + `theme.css` 语义样式 |
| 认证 | 服务端会话 + HttpOnly Cookie |
| 测试 | Vitest + Playwright + pytest |
| 本地启动 | `python start.py` 或根目录 `npm run dev` |

## 2. 目录分层

### 2.1 `apps/web`

职责：

- 承接公开页面路由
- 通过 Server Component 拉取目录数据
- 通过 Client Component 处理登录态和交互态
- 维护主题样式、组件和前端测试

当前真实页面：

- `/`
- `/tools`
- `/tools/[slug]`
- `/rankings`
- `/scenarios`
- `/scenarios/[slug]`
- `/auth`

### 2.2 `apps/api`

职责：

- 暴露目录、推荐、抓取、认证接口
- 管理数据库模型与迁移
- 承接会话认证和基础抓取调度

当前路由模块：

- `auth`
- `catalog`
- `ai-search`（AI 意图解析搜索，与 recommend 是两个独立系统）
- `recommend`（工具推荐）
- `chat`（流式 RAG 对话）
- `crawl`
- `parser`

### 2.3 `packages/contracts`

职责：

- 提供共享类型与种子数据
- 作为前后端契约收敛点

当前状态：

- 已存在共享类型
- 前端仍保留部分本地类型定义，尚未完全收口到共享包

## 3. 前端架构说明

### 3.1 渲染模式

- 首页和目录页通过 App Router 页面直接服务端取数
- 认证能力通过 `AppProviders -> AuthProvider` 在客户端恢复登录态
- `RouteFeedback` 负责路由切换反馈
- Header 根据登录状态动态展示“登录 / 注册”、用户名和退出入口

### 3.2 样式体系

- 全局入口：`app/globals.css`
- 主题沉淀：`src/styles/theme.css`
- 风格方向：玻璃质感、浅色工作台、较强语义类名复用

### 3.3 前端测试

- 单元测试：Vitest
- 页面测试：`HomePage.test.tsx`、`ToolsPage.test.tsx`
- 认证组件测试：`AuthCard.test.tsx`
- E2E：`smoke.spec.ts`、`visual.spec.ts`、`auth-checklist.spec.ts`

## 4. 后端架构说明

### 4.1 API 层

当前公开接口涵盖：

- 工具目录与详情
- 分类、榜单、场景
- AI 搜索（/api/ai-search，意图解析 + 工具筛选，含 AI Panel）
- AI 推荐（/api/recommend，批量返回推荐工具）
- Chat（/api/chat，流式 RAG 对话）
- 抓取任务入口
- 账号注册、登录、登出、当前用户

### 4.2 数据层

当前核心模型包括：

- 内容目录：`Tool`、`Category`、`Tag`、`Scenario`、`Ranking`
- 抓取链路：`CrawlJob`、`CrawlSnapshot`、`ToolUpdate`
- 认证链路：`User`、`UserSession`

### 4.3 调度与基础设施

- `apscheduler` 已接入应用生命周期
- 当前定时任务每 60 分钟触发一次抓取快照构建
- CORS 已对本地 Next.js 开发地址放开

## 5. 当前架构边界

已经落地：

- Monorepo 主体结构
- 前后端分离但同仓协作
- 真实 API 取数
- MySQL 模型与迁移
- 基础会话认证
- 基础单测与 E2E

尚未完全闭环：

- 前端未提供独立分类页
- 抓取任务仍是轻量实现，不是完整审核发布系统
- 共享类型仍有重复定义
- 后台管理端尚未落地
