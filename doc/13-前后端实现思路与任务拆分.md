# 13 前后端实现思路与任务拆分

## 1. 当前架构情况

当前项目已经形成稳定的 Monorepo 基线：

- `apps/web`：唯一公开前端，基于 Next.js App Router
- `apps/api`：唯一后端，基于 FastAPI + SQLAlchemy + Alembic
- `packages/contracts`：共享类型与样例数据

当前实现思路不再是“从旧 Demo 迁移”，而是“围绕现有正式基线继续补齐功能”。

## 2. 当前前后端职责边界

### 2.1 前端 `apps/web`

职责：

- 承接公开页面路由
- 服务端拉取目录类数据
- 客户端维护登录态
- 提供筛选、分页、认证交互

当前真实前端特征：

- 使用 App Router 页面取数
- 不依赖 `React Query`
- 不以浏览器侧 `mock API` 作为当前方案
- 登录态由 `AuthProvider` 驱动

### 2.2 后端 `apps/api`

职责：

- 提供目录、榜单、场景、推荐、抓取和认证接口
- 提供数据库模型与迁移
- 管理基础会话认证

当前真实后端特征：

- 目录接口已经稳定
- 认证接口已落地
- 抓取链路已有入口和调度，但仍是轻量实现

## 3. 当前仍需推进的任务

### Task 1：统一共享类型

- 收敛前端本地类型与 `packages/contracts`
- 清理重复定义
- 明确共享包的唯一来源地位

### Task 2：补齐目录与详情页面覆盖

- 继续完善工具详情、榜单、场景页的测试
- 明确各页面空态、错误态和超时态
- 视情况补充分类页前端实现

### Task 3：完善认证能力

- 在基础登录注册之上继续补充权限边界
- 明确哪些页面未来需要登录态保护
- 为后台预留角色模型

### Task 4：完善抓取与审核发布

- 将抓取快照、更新候选、审核发布串成闭环
- 明确 `tool_updates` 的审核流
- 补充接口和页面端的可观测性

### Task 5：补齐后台与运营能力

- 增加后台管理端
- 增加内容审核与发布界面
- 增加数据概览与运维页面

## 4. 当前任务优先级建议

1. 先统一文档、类型和现状口径
2. 再补测试覆盖和错误态
3. 再补抓取审核闭环
4. 最后再推进后台与商业化能力

## 5. 当前不应再继续做的事情

- 不再把旧 `Vite` 技术栈写成主线
- 不再围绕浏览器 `mock API` 设计新能力
- 不再把“登录注册不做”作为边界前提
