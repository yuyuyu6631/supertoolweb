# Agent Development Flow

## 1. 目标
把一个开发任务拆成 agent 能稳定执行的标准流程，尽量减少猜测、返工和一次性大改。

## 2. 核心原则

### 2.1 先确认边界
- 明确要改什么
- 明确不能改什么
- 明确验收标准
- 明确是否允许顺手重构

### 2.2 先摸底，再动手
- 先读相关代码、文档、配置和测试
- 先找到入口、调用链和关键约束
- 先确认当前真实实现，不靠记忆猜

### 2.3 分步推进
- 先实现最小可用改动
- 再补边界条件
- 再补测试和收尾
- 每一步都能独立验证

### 2.4 验证优先
- 每次改动后立即做局部验证
- 先看最小复现，再看完整回归
- 失败先定位根因，再继续改

## 3. 推荐输入

在开始前，尽量明确：
- 仓库路径
- 目标功能或问题
- 相关文件或模块
- 期望输出形式
- 验收标准

## 4. 执行步骤

### Step 1: 定位上下文
查找以下内容：
- 入口文件
- 相关模块
- 配置文件
- 现有测试
- 现有文档

### Step 2: 识别当前实现
确认这些问题：
- 真正的代码路径是什么
- 相关数据从哪里来
- 现有边界条件有哪些
- 当前实现的限制是什么

### Step 3: 制定计划
把任务拆成 3 到 5 步：
- 先做主路径
- 再补分支和异常
- 再补测试
- 最后整理输出

### Step 4: 实现改动
修改时遵守：
- 一次只做一个明确目标
- 不把无关重构混进来
- 不在未验证前大面积重写

### Step 5: 逐步验证
根据改动选择验证方式：
- 单元测试
- 集成测试
- 静态检查
- 本地运行或最小复现

### Step 6: 收尾整理
完成后检查：
- 是否有临时代码
- 是否有未处理的报错分支
- 是否需要补注释或说明
- 是否需要更新文档

### Step 7: 输出结果
最终汇报必须说明：
- 改了什么
- 怎么验证的
- 还剩什么风险
- 是否需要后续动作

## 5. 输出要求

推荐按下面顺序汇报：
1. 结果摘要
2. 实际改动
3. 验证结果
4. 残余风险
5. 后续建议

## 6. 禁止项
- 不先读代码就直接改
- 不把假设当事实写进实现
- 不跳过验证就宣布完成
- 不把一个问题拆成多个临时目录散落保存

## 7. 完成定义

满足以下条件才算完成：
- 需求已经落到代码或文档
- 相关验证已经执行
- 输出已经能交付
- 风险已经说明清楚
