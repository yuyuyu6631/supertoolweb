# 文档现状对齐工作流规范

## 1. 工作流目标

把仓库中的“当前实现类文档”更新到与代码一致的状态，并把“未来规划类文档”明确标注为未来态，避免历史方案和当前实现混写。

## 2. 基本原则

### 2.1 单一事实来源

- 当前代码是唯一事实来源
- 文档与代码冲突时，以代码为准
- 历史需求不能覆盖当前已采用实现，除非任务明确要求回退

### 2.2 文档分层

必须区分以下三类信息：

- 当前实现：已经在代码中落地
- 历史背景：旧方案、旧技术栈、旧路径
- 未来规划：尚未落地但保留方向价值

禁止把历史背景或未来规划写成当前实现。

### 2.3 先扫描，后改写

先识别事实，再编辑文档，不允许凭记忆大面积改写。

## 3. 输入要求

执行前至少应明确以下输入：

- 当前仓库根目录
- 需要更新的文档范围
- 当前代码基线的关键入口
- 用户的优先口径

默认优先口径：

- “现状为准”

## 4. 执行步骤

### Step 1：识别现状入口

先从以下位置建立事实基线：

- 前端入口：`apps/web/app/*`、`apps/web/package.json`
- 后端入口：`apps/api/app/api/router.py`、`apps/api/app/main.py`
- 数据模型：`apps/api/app/models/models.py`
- 前端类型：`apps/web/src/app/lib/*`
- 认证链路：`apps/api/app/api/routes/auth.py`、`apps/web/src/app/components/auth/*`
- 测试资产：`apps/web/src/app/**/__tests__/*`、`apps/web/src/e2e/*`、`apps/api/tests/*`

### Step 2：扫描文档偏差

在目标文档目录中搜索以下高风险关键词：

- 旧技术栈
- 旧路由
- 旧接口结构
- 旧页面边界
- 已失效的“不做”描述
- 与真实代码不符的数据来源说明

重点识别：

- 旧前端框架是否仍被写成现状
- mock API 是否仍被写成现状
- 已新增能力是否仍被文档标为“未实现”
- 已删除页面是否仍在文档中出现

### Step 3：给文档分类

把目标文档分成三组：

- 当前实现类：必须按代码重写
- 混合类：需要拆出“当前”和“未来”边界
- 未来规划类：保留方向，但必须加明确标识

### Step 4：统一改写口径

改写时遵循以下规范：

- 当前实现类文档只写代码里真实存在的内容
- 未来规划类文档开头必须显式标注“这是未来规划，不代表当前已实现”
- 不保留模糊语句，例如“可能”“预计”“大概是”作为现状描述
- 路由、接口、数据模型、依赖、测试应尽量列出真实名称

### Step 5：交叉核对

至少交叉核对以下内容：

- `README` 与技术架构文档一致
- 前端设计说明与实际页面路由一致
- 接口文档与后端路由定义一致
- 数据结构文档与模型/schema 一致
- 产品范围、验收标准与当前实现边界一致

### Step 6：失效信息复扫

修改完成后重新搜索高风险关键词，确认：

- 它们只出现在历史背景或未来规划语境
- 不再被写成“当前现状”

### Step 7：输出结果

输出必须包含：

- 本次更新了哪些文档
- 当前统一后的基线是什么
- 做过哪些校验
- 哪些内容仍属于未来规划

## 5. 推荐输出结构

推荐按以下顺序组织最终结果：

1. 结果摘要
2. 已更新文档
3. 当前统一基线
4. 已完成校验
5. 未覆盖风险或后续建议

## 6. 禁止项

- 不要把旧技术栈继续写成当前实现
- 不要把未来规划偷偷混入现状说明
- 不要只改一篇索引文档而放过正文冲突
- 不要凭主观猜测补接口、补页面、补模型
- 不要在未核对代码的情况下写“已完成”

## 7. 完成定义

满足以下条件才算完成：

- 目标文档已全部更新
- 关键文档之间口径一致
- 高风险旧关键词已被清理或重标
- 当前实现和未来规划已清晰分层
- 最终输出中说明了本次核对依据
